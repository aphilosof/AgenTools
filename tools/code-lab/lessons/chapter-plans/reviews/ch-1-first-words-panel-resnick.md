# Panel Review — Ch.1 First Words — Mitchel Resnick (Scratch, MIT Media Lab)
**Verdict:** Needs fixes

## Findings (from my lens)

1. [BLOCKER] Plan vs. implementation mismatch — the chapter plan lists indexing/slicing (`s[0]`, `s[-1]`, `s[1:3]`, etc.), `.join()`, f-strings (including `:.2f`), and `round()` as owned Ch.1 concepts taught in L1.5. None of these appear in the actual c1.js lesson 1.5 implementation. The written lesson covers only concatenation, repetition, `.upper()`, `.lower()`, `.strip()`, `.split()`, and `len()`. Similarly, `sep` and `end` are listed as L1.1 owned concepts in the plan but were explicitly "moved to Chapter 2" per c1.js's own comment block, and appear in c2s5. A chapter plan that claims ownership of concepts the implementation never delivers is not a plan — it is fiction. Before any prose is added to the plan, the owned-concepts list must be reconciled against what c1.js actually teaches. Either the implementation is incomplete (the lesson needs to be finished) or the plan claims ownership of concepts that belong elsewhere. Fix: audit every owned concept in the plan against c1.js, then either (a) write the missing subsections in c1.js, or (b) reclassify the concepts to the chapter that actually teaches them, and update the concept spine in CURRICULUM-STRUCTURE.md accordingly.

2. [MAJOR] L1.7 strand lesson gives students no creative latitude — the two exercises (turtle square, three-note motif) both terminate with "write a program that draws two squares of different sizes" and "write-from-scratch a two-bar motif." These are well-scaffolded, but the prompt is fully specified: exact output is described, the goal is given. Compare this to the Ch.0 strand lesson where students chose their own four pitches from a bank ("multiple correct answers"). Ch.1's strand lesson is the first moment in the chapter where the student could make something that feels genuinely theirs — a drawing, a melody — and the plan prescribes the exact outcome. The motif exercise in particular (`root + 5` for bar 2) is not expression; it is arithmetic in disguise. Fix: restructure at least one strand exercise as a genuinely open prompt. For the turtle: "Draw anything that is symmetric" — the checker verifies that `right()` angles sum to a multiple of 360, not that the output is two specific squares. For the motif: "Write a four-note pattern using `root` and offsets you choose — the only rule is that the last note uses `root + 12` so it resolves one octave higher." This gives intrinsic motivation — the student hears their own musical idea — without removing the scaffolding. A checker can verify structure (four `play()` calls, last note = root + 12) without dictating content.

3. [MAJOR] The Chapter Challenge's creative exercises are all fully specified — there is no moment where a student makes something their own. Items 1 through 9 all have a correct answer that is either a literal value or a precisely described program. Item 9 (transposing motif) is the closest to expressive — reading a root note from `input()` and playing four offsets — but the offsets (+0, +2, +4, +7) are given. This is the chapter's capstone. If a student finishes the chapter having never made a choice about what their program *expresses*, they have learned Python syntax but not that code is a creative medium. Fix: add one genuinely open capstone item after item 9. Something like: "Using what you know, write a short program that generates any greeting you want, for any person, with their name and a message that uses at least one f-string and one string method." No prescribed output. Checker verifies structure (input() present, f-string present, string method called), not content. Wide walls: thousands of valid solutions. This is the wide-walls moment the chapter needs.

4. [MINOR] L1.6 mini-project (age-in-days calculator) — reads name and birth year, computes age in years and days, prints an f-string summary. Problem: f-strings are claimed as a Ch.1 owned concept in the plan (L1.5) but are absent from c1.js. The mini-project's solution therefore uses a feature the lesson never taught. If f-strings remain in Ch.1 scope and get written into c1.js, this is fine. If f-strings are deferred, the mini-project's solution approach needs to use string concatenation + `str()` instead, and the f-string mention in the mini-project description needs to be removed. As it stands, the plan is internally inconsistent: L1.5 lists f-strings as owned, L1.6 uses them in a mini-project, and c1.js teaches neither. This is a downstream consequence of Finding 1.

5. [MINOR] L1.5 "String building" synthesis subsection — the plan says "combine indexing, slicing, `.split()`/`.join()`, and f-strings to build multi-part outputs; a name-mashup generator." Since indexing, slicing, f-strings, and `.join()` are all missing from c1.js, this synthesis subsection describes an exercise that requires four untaught tools. Even if the plan intends these to be written before the lessons are built, the synthesis subsection is currently an orphan — it cannot exist without the earlier subsections that introduce its inputs. The synthesis exercise (name-mashup, formatted report) is an excellent creative idea. But it only works as expression if students have already internalized the building blocks. Fix: as part of resolving Finding 1, ensure the synthesis subsection is the *last* subsection in L1.5 and that every tool it uses has its own prior subsection in the same lesson.

6. [MINOR] Truthiness scope note in L1.2 — the plan includes: "Scope note for students: 'For the types we've seen so far, those are all the falsy values — collections like lists will add a few more in a later chapter.'" This is appropriately foreshadowing, but note that c1.js's actual implementation of the bool/truthiness content (L1.2) omits `0.0` from its falsy-values example — `bool(0)` is shown but `bool(0.0)` is not. The quiz plan also does not include `0.0` in the falsy-identification question. The plan's text does mention `0.0` as falsy (correct per Python docs), but the implementation drops it. The quiz asking "which of five values is falsy?" should include `0.0` as a case to ensure students know both numeric falsy values. This is a small but correctable omission in the quiz design.

7. [MINOR] `sep` and `end` are listed as L1.1 owned concepts in the chapter plan but were moved to Chapter 2 (c2s5) in the actual implementation, where they appear in the context of `for` loops and `range()`. The chapter plan still claims them under L1.1. This is not just a metadata error — c2s5 is the right chapter for `sep` and `end` because seeing `print(i, end=" ")` inside a loop is the natural moment of need. Putting them in L1.1 would be answering a question students have not yet asked. The plan should be updated to reflect what is built: `sep` and `end` are owned by Ch.2 (L2.5), not Ch.1 (L1.1).

## Continuity notes

**Well-woven from c1–c3 (what the plan gets right):**
- The plan correctly uses Ch.0 recall: L1.1 warm-up draws on "what does a sequence of exact steps look like?" and L1.3's warm-up retrieves `type(3.14)` from L1.2. These are concrete, single-fact retrievals, not vague "remember Chapter 1" gestures. This is the right pattern.
- The plan does not re-teach Ch.0 concepts (sequence, algorithm, debugging idea). It notes them as prior context without re-explaining them, which is correct spiraling.
- The stepper thread from L1.2 onward (predict-the-variable-table) is the right tool for making the notional machine visible. c1.js's actual L1.3 does this correctly: the reassignment example (`score = 10; score = score + 5`) maps directly to what the stepper shows. The plan reinforces this pattern and c1.js delivers it.
- The plan's handling of `bool` and `None` in L1.2 is appropriately complete and correctly defers boolean operators to Ch.2. c1.js's actual L1.2 teaches `bool()` and truthiness without over-reaching into `and`/`or`/`not`, which Ch.2 owns.
- c2.js and c3.js build on Ch.1 without re-teaching it. c2s1 opens with a hook `if` example and immediately moves to comparisons — it assumes `bool` is known, variables are known, `print` is known. That continuity assumption is correctly calibrated.

**Missing or underprepared:**
- f-strings appear exactly once in c2.js (line 617, inside a loop: `print(f"i = {i}")`) with no introduction and no teaching. They are used as if the reader already knows them — which the plan claims they do (Ch.1 owned), but which the implementation never delivers. Any student who has only worked through c1.js as written will hit this f-string in a loop body in Ch.2 with no preparation. Fix: f-strings must be either formally taught in a c1.js subsection before Ch.2 uses them, or their first appearance in c2.js must be introduced with a brief inline explanation. Leaving this gap means the spiral breaks at the exact moment the course starts applying the concept.
- Indexing and slicing are listed as Ch.1 owned but used nowhere in c2.js or c3.js either. This means the gap identified in Finding 1 is not currently plugged downstream. If indexing/slicing are genuinely in scope for this course, they need to be taught somewhere before they are needed. The plan places them in Ch.1; c2.js and c3.js appear to have been written without assuming them.
- The L1.6 mini-project (age-in-days) integrates all of 1.1–1.5, which is the right design. But in c1.js's actual implementation, the age-greeter exercise (c1s6, rung 6) is simpler: name + age + "Hi Alex, next year you will be 12." The mini-project as described in the plan is more ambitious (birth year, current year variable, days computed). Neither is wrong, but they differ. Whoever writes the missing content should know that c1s6 as implemented is a rung-6 starter, not a multi-step mini-project.

**Re-taught by accident:**
- None found. The plan does not re-explain Ch.0's sequence/algorithm/debugging concepts as Python concepts — it correctly treats them as assumed context and gives them Python syntax.

## What's strong

- **The through-line is genuinely coherent.** "A program is a sequence of instructions that acts on values — every value has a type, and types decide what you can do." This is a real mental model, not a tagline. The HTTLACS ordering (values before variables, operators before strings, `input()` last) is the correct sequence for building that model incrementally, and the plan follows it faithfully.

- **The misconceptions table is the strongest in the course so far.** The plan correctly identifies and distinguishes `^` (bitwise XOR) as producing a silent wrong answer — not a SyntaxError — and marks this as harder to catch than a traceback. That distinction is real and important, and it is at exactly the right difficulty level for a strong 11-year-old who already trusts that error messages will catch their mistakes. This is sophisticated pedagogy.

- **The rung ramp is honestly calibrated.** L1.1 opens at rung 1, L1.6 closes at rung 6 write-from-scratch, and the progression within each lesson fades scaffolding systematically. The plan does not claim write-from-scratch as a default by the end of Ch.1 — it correctly reserves that as the "Chapter 3 milestone" per CURRICULUM-STRUCTURE.md, while still getting students to blank-editor work by the chapter's end for concepts they've fully internalized. This is the right approach.
