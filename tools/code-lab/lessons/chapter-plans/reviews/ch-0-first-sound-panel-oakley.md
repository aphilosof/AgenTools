# Panel Review — Ch.0 First Sound — Barbara Oakley (Learning How to Learn)
**Verdict:** Solid

## Findings (from my lens)

1. [MAJOR] L0.4 warm-up — The free-text reflection ("Why is decomposing 'get home' better than one instruction?") is listed as "not auto-checked — used to prime the lesson's theme." This is the only warm-up in the chapter that is neither auto-checked nor designed as a concrete retrieval cue. Every other warm-up in Ch.0 asks for a specific, verifiable answer; this one invites open reflection. Reflection is not retrieval practice. The student may fluently *describe* the benefits of decomposition (they just read about it in L0.3) without having actually retrieved the concept from memory — this is a textbook illusion of competence. The student feels like they know it because the words are fresh. Fix: replace the free-text reflection with a concrete forced-choice retrieval item, e.g. "A friend wants to write one instruction: 'go home.' They break it into four sub-steps instead. Which benefit does that give them? (A) fewer steps / (B) each sub-step is small enough to solve on its own / (C) the algorithm runs faster / (D) decomposition is only useful for big problems." This retrieves the same concept but forces genuine recall rather than recognition of recently-read prose.

2. [MAJOR] Retrieval set spacing — The five recap retrieval questions are described as appearing "as warm-ups in Ch.1 lessons (one per lesson, rotating)." Ch.1 has seven lessons; the plan says "one per lesson, rotating," which implies each question appears once across Ch.1 and then stops. That is a single retrieval trial per concept, spaced by only a few lessons. The research on the spacing effect is unambiguous: a single delayed retrieval trial after a few days helps but two or three trials at expanding intervals (Ch.1, Ch.2, Ch.4) are dramatically more effective for long-term retention. The plan sets up retrieval practice but does not build in the *re-retrieval* that makes spacing work. Fix: explicitly mark each of the five retrieval items with a spiral schedule — "resurface in Ch.2 warm-up" and "resurface in Ch.4 warm-up" — and record this in the concept spine. Otherwise the team building Ch.2 and Ch.4 has no contract to pick these up, and they will be dropped.

3. [MINOR] L0.5 warm-up — arithmetic discrepancy. The warm-up asks: "A loop says `repeat 4 times: play(60), sleep(0.5)`. How many notes play in total?" The answer given is 4. This is correct (one `play()` per iteration, 4 iterations). However, in L0.4 exercise 3, the analogous trace question is: "Pseudocode loop `repeat 3 times: play(60), play(64), sleep(0.5)`. How many notes play in total?" The answer there is 6 (two plays per iteration, 3 iterations). The L0.5 warm-up is a retrieval cue for L0.4's loop-tracing, but it uses a simpler case (one play per iteration) while L0.4's exercise used a harder case (two plays). That is backwards — the warm-up should be at least as demanding as the original exercise, otherwise it tests a different skill (counting single-body iterations rather than multi-body iterations) and the off-by-one misconception from L0.4 is not re-tested. Fix: change the L0.5 warm-up to match the harder form: "A loop says `repeat 4 times: play(60), play(64), sleep(0.5)`. How many notes play in total?" Answer: 8. This retrieves the same multi-body counting skill the lesson introduced.

4. [MINOR] Cognitive load in L0.3 — three new things at once. Lesson L0.3 introduces: (1) Parsons/arrange as a *task type* (students have not done drag-and-drop before), (2) decomposition as a *concept*, and (3) a grid-robot *domain analogy* (students may not have seen a grid-navigation abstraction). The worked-example sequence handles this reasonably — it shows a fully correct Parsons drag and then lets the student restore it before any exercises — but the lesson carries a non-trivial element-interactivity load for learners encountering all three simultaneously. The plan does not acknowledge this. This is not a blocker because the rung is low (rung 2, no syntax), but the lesson narrative should explicitly say: "This is a new type of exercise — here is how it works" before the first Parsons task. Without that orienting sentence the student may spend cognitive resources figuring out the interface rather than processing the concept. The plan implies this is obvious; for an 11-year-old it is not. Fix: add an explicit "How this works" orientation panel before the first Parsons task in L0.3.

5. [MINOR] Chapter Challenge exercise 3 (loop Parsons) uses a `repeat 4 times` block. The loop-count changes from the worked examples and exercises in L0.4, which used 3 and 4. The challenge is consistent — but note that when changing the iteration count from examples students have seen, there is a risk of pattern-matching rather than genuine tracing. If students recall "the loop exercise used 3 × 2 = 6, so the answer is 6," they will mis-transfer. This is low-risk but worth a note to the lesson author: in the challenge, explicitly prompt the student to trace step by step rather than compute from memory ("Work it out one step at a time — don't just multiply").

6. [MINOR] L0.6 (strand-application lesson) introduces no new retrieval practice. The warm-up is a quick recall of the "bug" definition from L0.5 — one item, one concept. As the final content lesson before the challenge and quiz, this is the highest-value moment for interleaved retrieval of *all* the chapter's concepts before the gate. The warm-up should be a brief interleaved block: one question from L0.2 (algorithm precision), one from L0.3 (decomposition), and the bug question from L0.5 — three items, thirty seconds. This creates the pre-quiz retrieval effect documented by Roediger and Karpicke (2006): retrieval before assessment improves the assessment itself and produces better long-term retention than a single warm-up item. Fix: expand the L0.6 warm-up to three rotating short-answer items drawing from L0.2, L0.3/L0.4, and L0.5.

7. [MINOR] The Chapter Quiz at the end awards a pass at "4/5 auto-graded questions correct." Question 5 (free write: explain algorithm) is not auto-graded but is "presented" and "surfaces in coach mode." For spaced-repetition hygiene, Question 5 should also generate a retrieval item that re-surfaces in Ch.1 — not for grading, but because generating an explanation is one of the most powerful encoding strategies known (the generation effect). The plan sends it to coach mode and effectively ends it there. Fix: retain the free-write but explicitly schedule it as an item that re-appears in a Ch.1 reflection prompt ("In your own words, what is an algorithm? In Ch.0 you wrote: [student's previous answer]. How would you say it differently now?"). This closes the loop.

## Continuity notes

**Well-woven:**
- The recall architecture within Ch.0 is correctly structured: every lesson from L0.2 onward opens with a warm-up that retrieves from the preceding lesson. This is textbook spaced retrieval at a short interval — appropriate for new concepts in their first exposure week. The chain is clean: L0.2 recalls L0.1 → L0.3 recalls L0.2 → L0.4 recalls L0.3 → L0.5 recalls L0.4 → L0.6 recalls L0.5. No gaps.
- The five Codex retrieval questions at the chapter recap are well-chosen. They are not definitional lookups ("what is a loop?") but application and trace questions ("how many seconds does this loop produce?"). Application retrieval is far more effective than definition retrieval for transfer.
- The rung-1/rung-2 constraint correctly avoids syntax overload. Ch.0 students are building conceptual schemas, not procedural syntax; keeping them separate prevents the common failure mode where syntax anxiety blocks conceptual uptake.

**Missing or underprepared (relevant to learning science):**
- There is no explicit interleaving across concepts *within* the chapter's exercises. Every lesson practices only its own concept: L0.3 exercises are all about sequence/decomposition; L0.4 is all about loops and branches. Interleaving — mixing problem types from different concepts within a single exercise set — is one of the two most research-validated spacing techniques (alongside distributed practice). The Chapter Challenge does interleave, which is good, but that is one session at the end. In L0.5 and L0.6, the exercises could cheaply mix in a sequence-ordering item or a loop-count trace to interleave concepts already taught. The plan currently uses blocked practice within lessons, then one shot of interleaving at the challenge.
- The spiral schedule for Ch.0 retrieval items into Ch.1 is described ("one per lesson, rotating") but is not given a concrete implementation contract. Who decides which question appears in which Ch.1 lesson? The plan says "rotating" but does not assign. This will cause the retrieval items to be omitted when Ch.1 is authored, because Ch.1's authors will treat them as Ch.0's concern. Fix: assign each of the five retrieval items to a specific Ch.1 lesson number in the plan itself.

**Re-taught by accident:**
- None found. Ch.0 correctly does not re-explain anything owned by Ch.1. The `play()`/`sleep()` API is treated as opaque by design, which is the right call — teaching it as Python syntax here would front-load Ch.1's function-call concept and cause interference rather than preparation.

## What's strong

- **The spaced retrieval architecture within the chapter is clean.** Each lesson's warm-up retrieves exactly one item from the prior lesson at a short interval. This is the right cadence for initial encoding of new material — the plan correctly does not try to recall across multiple lessons in early warm-ups, which would overload a student who has not yet consolidated the intervening concepts.
- **The chunking of CT concepts is exemplary.** Sequence (L0.1), algorithm/precision (L0.2), sequence applied + decomposition (L0.3), loop + branch (L0.4), debugging (L0.5) — each chunk is exactly one new idea, grounded in a familiar domain analogy before the CS name appears. The dependency order means no lesson asks students to hold more than one new concept in working memory at a time. This is textbook worked-example effect design.
- **The sound hook is cognitively ideal as a low-stakes retrieval anchor.** Every subsequent lesson returns to `play()`/`sleep()` examples. This means the sound domain acts as a retrieval-friendly context that students can mentally revisit when a concept is fuzzy — "remember that melody exercise" serves as a concrete memory cue. The through-line is not just motivating; it is functionally useful for memory retrieval across the chapter.
