## Subsection S1 — "`print` shows output"  (concepts: print, string-literal, expression-as-arg)

**concept.md** (expand to exemplar depth): A program shows nothing until you tell
it to; `print` is that instruction. It takes whatever is inside its brackets — the
[[argument]] — and displays it. Text must sit inside quotation marks (single or
double), which makes it a [[string]]; the quotes only mark where the text starts
and ends and are **never shown**. Each `print` writes its own line, so several
prints stack in order. A `print` can take more than one argument separated by
commas, and it puts a single space between them. The brackets do not have to hold
plain text, either — they can hold a small calculation, and `print` shows the
**result** of that calculation, not the calculation itself. Use
`{{more:Single or double quotes?|Python accepts both 'hi' and "hi"; pick one so the
other can appear inside it: "it's fine".}}`.

**examples** (5, easy → harder; all verified 3.13.7):
1. `print("Hello!")` → `expect:{type:"output", value:"Hello!"}`
   (simplest visible program; the quotes mark the text but do not appear)
2. `print("Code")\nprint("Lab")` → `expect:{type:"output", value:"Code\nLab"}`
   (two prints stack in order)
3. `print("The answer is", 42)` → `expect:{type:"output", value:"The answer is 42"}`
   (multi-argument; one space inserted; first non-string argument — frame as
   "print can show a number too", do NOT teach types here → 1.2)
4. `print(2 + 3)` → `expect:{type:"output", value:"5"}`  **(settled decision 1)**
   Expressions as arguments. Framing: the brackets can hold a little calculation,
   and `print` shows its **result**, not the text `2 + 3` — so `print(2 + 3)` prints
   `5`, while `print("2 + 3")` would print the words `2 + 3`. Keep it to "print
   evaluates what's inside, then shows the answer"; do NOT open number *types* or
   operator precedence (1.2 / 1.4 own those). Read-only/runnable; no exercise.
5. `print("Score:", 7 * 3)` → `expect:{type:"output", value:"Score: 21"}`
   (the harder synthesis example — a calculation *and* a label in one multi-argument
   print: text first, then the result of `7 * 3`, with one space between. Reinforces
   "the brackets can hold a calculation" in a second shape. Still type-free.)

## Subsection S2 — "Combining text: `+` to join, `*` to repeat"  (concepts: string-concat, string-repeat)  **NEW**

This is the richer content the thin draft was missing. Everything here is **string
with string, shown working** — no `str` type, no `str()`, no `str + int` error, no
"types" vocabulary. Build the foundation now; the type machinery is 1.2's job.

**concept.md** (expand to exemplar depth): You have printed single pieces of text;
now combine them. Two operators do this. `+` performs [[concatenation]] — it glues
two strings end to end into one longer string: `"foot" + "ball"` is `"football"`,
and `print("Code" + "Lab")` shows `CodeLab`. Notice there is **no space** — `+`
joins exactly what you give it and nothing more, so if you want a gap you add it
yourself as a space in quotes: `"Code" + " " + "Lab"` shows `Code Lab`. This is the
key difference from the comma you met in the last subsection: a comma between two
arguments tells `print` to insert one space, while `+` glues with none.
`print("hot" + "dog")` shows `hotdog`; `print("hot", "dog")` shows `hot dog`.

The other operator is `*`. Used between a piece of text and a whole number it
**repeats** that text: `"ab" * 3` is `"ababab"`, and `"-" * 10` is a row of ten
dashes — a quick way to draw a divider line or a banner without typing every
character. You can even combine the two: `"=" * 3 + " MENU " + "=" * 3` builds a
little title bar, `=== MENU ===`. Keep `{{more:Why no space from +?|+ is literal:
it joins the exact characters in each string. A comma is print's own helper that
adds one space between arguments. Two different tools — choose the comma when you
want spacing for free, and + when you want to control every character yourself.}}`.

(Author note — boundary, do NOT cross: stay on text-with-text. Do **not** show
`"Score: " + 5` or `str(5)`; the `str + int` `TypeError` and conversion belong to
1.2/1.5. Do **not** name "types". `*` is shown only as *text × whole number*; the
`str * float` failure is 1.2's.)

**examples** (5, easy → harder; all verified 3.13.7):
1. `print("Code" + "Lab")` → `expect:{type:"output", value:"CodeLab"}`
   (the join is exact — no space appears)
2. `print("Code" + " " + "Lab")` → `expect:{type:"output", value:"Code Lab"}`
   (add your own space-in-quotes to get a gap)
3. `print("ab" * 3)` → `expect:{type:"output", value:"ababab"}`
   (repeat a piece of text a whole number of times)
4. `print("Ho" * 3 + "!")` → `expect:{type:"output", value:"HoHoHo!"}`
   (a playful mid-difficulty case: repeat, then join an extra piece — `*` happens
   before `+`, so `"Ho"` repeats first and `"!"` joins after)
5. `print("=" * 3 + " MENU " + "=" * 3)` → `expect:{type:"output", value:"=== MENU ==="}`
   (the hardest synthesis example: combine repeat and join to build a title bar)

## Subsection S3 — "Comments: notes Python ignores"  (concept: comment)

Deliberately lighter middle beat (relaxed lesson-level ramp, decision 4). Keep
rungs non-decreasing *within* S3; do NOT pad it to the others' ceiling.

**concept.md**: A comment starts with `#` and runs to the end of the line; Python
skips everything after it. Comments are notes for the people reading the code — they
never run. A `#` can sit on its own line above the code it explains, or ride at the
end of a line of code (an *inline* comment). Because Python ignores whatever follows
`#`, you can also "comment out" a real line — put a `#` in front of it to switch it
off without deleting it. Comments change nothing about what the program does; they
only change how easy it is for a human to read.

**examples** (3; verified 3.13.7):
1. `# This program greets you\nprint("Hi")` → `expect:{type:"output", value:"Hi"}`
2. `print("Go")  # this part is ignored` → `expect:{type:"output", value:"Go"}`
3. `# print("secret")\nprint("shown")` → `expect:{type:"output", value:"shown"}`

## Subsection S4 — "Shaping output: `sep` and `end`"  (concepts: sep, end)

The hardest subsection → ramp toward write.

**concept.md**: When `print` gets several arguments it puts one space between them
by default — that is why `print("a", "b")` shows `a b`, and (tying back to S2) why a
comma behaves differently from `+`. You can change that in-between text with
[[separator]] (`sep`), and change the text printed *after* everything with [[end]].
By default `end` is a newline — the invisible character that drops to the next line
— which is why each `print` starts fresh below the last. Set `end` to something
else, even an empty `""`, and the next `print` carries on where this one stopped, so
two prints share a line. You write both as `name=value` inside the brackets, like
`sep="-"`; that style is called a **keyword argument**, and you will meet many more
later. For now: `sep` goes *between* the things, `end` goes *after* them all.

**examples** (4; verified 3.13.7):
1. `print("a", "b", sep="-")` → `expect:{type:"output", value:"a-b"}`
2. `print("done", end="!")\nprint("next")` → `expect:{type:"output", value:"done!next"}`
3. `print("red", "green", "blue", sep="/")` → `expect:{type:"output", value:"red/green/blue"}`
   (sep works for any number of arguments — it goes between each pair)
4. `print(2024, 11, 30, sep="-")` → `expect:{type:"output", value:"2024-11-30"}`
   (a practical use — building a date with one separator. The numbers here are not
   text, but `sep` is `print`'s own job: it joins the printed items with the chosen
   separator. Frame as "sep sits between whatever you print"; do NOT open types.)

**exercises** (3; ramp 3→5→6):

  `sep="-"` inside the brackets.

  `starter`: `print("red", "green", "blue", sep=)  # fill in the separator\n`

  (verified 3.13.7: the empty `sep=)` raises `SyntaxError: expected argument value
  expression` until filled — registers SyntaxError honestly)
  `check`: `{type:"output", value:"red/green/blue"}`
  `solution`: `print("red", "green", "blue", sep="/")\n` · `errorClasses`:

## Notes — verified facts (state as facts in prose, not approximations; all 3.13.7)
1. Multi-arg spacing is exactly **one** space (`print("A","B")` → `A B`); `sep`
   default is a space, `end` default a newline.
2. `+` between two strings adds **no** space: `print("Code" + "Lab")` → `CodeLab`;
   `print("Code" + " " + "Lab")` → `Code Lab`.
3. `*` repeats text by a **whole number**: `"ab" * 3` → `ababab`; `"Ho" * 3 + "!"`
   → `HoHoHo!`; `"-" * 10` → `----------`; `"=" * 3 + " MENU " + "=" * 3` →
   `=== MENU ===`.
4. The comma-vs-`+` contrast: `print("hot" + "dog")` → `hotdog`;
   `print("hot", "dog")` → `hot dog`.
5. S1 E2 fix: real 3.13.7 traceback is `NameError: name 'Hello' is not defined` —
   quote it exactly.
6. `print(2 + 3)` → `5`; `print("2 + 3")` → `2 + 3` (the contrast S1 ex 4 relies
   on); `print("Score:", 7 * 3)` → `Score: 21` (S1 ex 5).
7. S4 E2 stub `sep=)` raises `SyntaxError: expected argument value expression` on
   3.13.7 — `starterExpectation:"raises:SyntaxError"` is honest.
8. S2 E1 Parsons distractors on 3.13.7: `print("=" + 12)` raises
   `TypeError: can only concatenate str (not "int") to str`; `print(MENU)` raises
   `NameError: name 'MENU' is not defined`. Both are genuine wrong answers; neither
   is taught content.
9. `print(2024, 11, 30, sep="-")` → `2024-11-30` (S4 ex 4 — sep joins the printed
   items; this is `print`'s own behaviour, not `str + int`, so it stays inside the
   I/O boundary).
10. `print()` with no arguments prints a blank line (available, not required).
11. **Boundary held:** nothing in S2 introduces the `str` type, `str()`, the
    `str + int` `TypeError`, `str * float`, or "types" vocabulary. `"Score: " + 5`
    (which raises `TypeError`) is deliberately **not** used as taught content — that
    lives in 1.2.
