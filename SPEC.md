# AgenTools — Repository Specification

## Purpose

A central repository of reusable Claude Code **agents**, **skills**, and **tools** that can be referenced from any Claude Code project via CLAUDE.md includes. This is a living library — always consumed directly from this repo, never copied into target projects.

## Principles

- **Central, not distributed**: All projects reference this repo directly. Nothing is copied or installed.
- **Cross-dependencies allowed**: Items can depend on other items in this repo (always at HEAD — no internal version pinning).
- **Language-agnostic**: Items may target specific languages/frameworks, but the repo structure and tooling are language-neutral.
- **Per-item versioning**: Each item carries its own semantic version in its manifest. Versions track the external-facing contract.
- **Standardized structure**: Every item follows a template with consistent layout and metadata.

## Folder Structure


### Category Examples

Categories are flexible and grow organically. Examples:
- `git/` — git-related operations
- `testing/` — test generation, running, analysis
- `refactoring/` — code transformation and cleanup
- `docs/` — documentation generation
- `devops/` — CI/CD, deployment, infrastructure

## Item Types

### Agents
Autonomous sub-processes that handle complex, multi-step tasks. Defined by a system prompt and optionally tool configurations.

### Skills
Reusable prompt-based capabilities invoked via slash commands or CLAUDE.md references. Typically a prompt file with optional supporting logic.

### Tools
Discrete utilities — scripts, MCP server definitions, or function specs — that extend Claude Code's capabilities.

## Manifest Format (manifest.yaml)

Every item MUST have a `manifest.yaml` at its root:

```yaml
name: my-tool-name
type: agent | skill | tool
version: "1.0.0"
description: >
  One-paragraph description of what this item does.

tags:
  - git
  - automation

compatibility:
  languages:
    - python
    - typescript
  frameworks:
    - fastapi
  notes: "Requires Node 18+ for the MCP server component"

dependencies:
  - tools/git/commit-formatter   # path to another item in this repo

usage:
  examples:
    - description: "Basic usage"
      snippet: |
        Include in your CLAUDE.md:
        @/path/to/AgenTools/tools/git/commit-formatter/README.md

    - description: "As part of a workflow"
      snippet: |
        Use the commit-formatter tool after staging changes.
```

### Required Fields
- `name`, `type`, `version`, `description`

### Optional Fields
- `tags`, `compatibility`, `dependencies`, `usage`

## Registry (registry.yaml)

Auto-generated by `scripts/generate_registry.py`. Provides a flat, browsable index:

```yaml
generated_at: "2026-03-16T12:00:00Z"
items:
  - name: commit-formatter
    type: tool
    version: "1.0.0"
    path: tools/git/commit-formatter
    description: Formats commit messages to follow conventions.
    tags: [git, formatting]
    compatibility:
      languages: [any]
  - name: test-runner
    type: agent
    version: "0.2.0"
    path: agents/testing/test-runner
    description: Runs and analyzes test suites.
    tags: [testing]
    compatibility:
      languages: [python, typescript]
```

The registry script is run manually or via a git hook after changes. It reads all `manifest.yaml` files and outputs `registry.yaml`.

## Consumption from Other Projects

In a target project's `CLAUDE.md`, reference items like:

```markdown
# Tools
@/Users/alonphilosof/Dropbox/AgenTools/tools/git/commit-formatter/README.md

# Agents
@/Users/alonphilosof/Dropbox/AgenTools/agents/testing/test-runner/README.md
```

Each item's README serves as the interface contract — it contains the instructions Claude needs to use the item.

## CLAUDE.md (This Repo)

The repo's own CLAUDE.md acts as a contributor guide, teaching Claude how to:
1. Create new items using templates
2. Follow manifest conventions
3. Update the registry after changes
4. Maintain versioning discipline

## Versioning Rules

- Use semantic versioning: `MAJOR.MINOR.PATCH`
- Bump PATCH for bug fixes and minor tweaks
- Bump MINOR for new capabilities that don't break existing usage
- Bump MAJOR for breaking changes to the item's interface
- Internal dependencies always reference HEAD (no version pinning between items in this repo)
- Version changes should be noted in the item's README under a changelog section
